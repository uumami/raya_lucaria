# Best Practices

## Workflow

### Always validate before commit

```bash
docker compose up validate
```

This runs `validate_content.py` which checks for:
- Unclosed components
- Duplicate component IDs
- Invalid date formats
- Missing required attributes
- Broken internal links

### Test with Docker build

This repo serves as both the framework and a working example course. Test framework changes with:

```bash
docker compose up build
```

## Common Operations

### Add a new chapter

1. Create directory `clase/NN_chapter_name/` with appropriate numeric prefix
2. Create `clase/NN_chapter_name/00_index.md` with `layout: layouts/chapter.njk`
3. Add content files `01_topic.md`, `02_topic.md`, etc.
4. Rebuild -- the chapter appears automatically in navigation

### Add a component to a page

Insert the component syntax in any `.md` file:

```markdown
:::homework{id="unique-id" title="Task Name" due="2026-03-01"}
Instructions here.
:::
```

### Modify navigation

Navigation is auto-generated from the file hierarchy. To change nav order, rename files with different numeric prefixes. To hide content, use the `??_` prefix.

### Add a new theme

1. Copy an existing theme CSS file in `src/eleventy/src/css/themes/`
2. Change the CSS custom property values (hex colors)
3. Add the theme name to `available` list in `glintstone.yaml`

## Danger Zones

- **Never edit `_data/*.json` manually** -- These files are generated by preprocessing and will be overwritten on next build
- **Never edit files in `_site/`** -- This is the build output directory
- **Never edit files in `clase-stage/`** -- This is a temporary staging directory created during build
- **Be careful with `00_index.md`** -- Deleting it breaks the chapter's nav entry

## Code Style

### Thin orchestrator pattern

`.eleventy.js` is intentionally kept short (~60 lines). All logic lives in `lib/` modules:
- `lib/markdown.js` -- markdown-it configuration
- `lib/components.js` -- Component registry and rendering
- `lib/filters.js` -- Nunjucks filters
- `lib/collections.js` -- Collection definitions
- `lib/transforms.js` -- HTML transforms

### Python conventions

- Single `utils.py` pattern -- shared logic lives in one place
- Dataclass schemas in `schemas/config.py`
- Logger via `preprocessing.logger.get_logger()`

### File naming

- Python: `snake_case.py`
- JavaScript: `camelCase.js` for lib modules
- Templates: `kebab-case.njk`
- Content: `NN_snake_case.md` with numeric prefix
